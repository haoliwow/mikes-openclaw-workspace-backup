## 🔴 必問 3 題（所有職缺通用）

### **Q1【關鍵風險】台北富邦 4 個月為什麼離職？**

**背景**：
- 2024-09~2024-12 台北富邦銀行（業務中台架構師經理）
- 2024-12~至今 國泰金控（資深工程師）
- **只待 4 個月就轉換**

**提問方式**（自然開場）：
> 「我看妳的履歷，在台北富邦業務中台待了 4 個月就轉到國泰金控。這期間發生了什麼？是集團內轉調，還是職位本身不太適合？」

**好答案徵兆** ✅：
- 「台北富邦是試驗性轉調，後來集團決定在國泰金控強化這個方向，所以回歸」（集團內流動，合理）
- 「一開始是臨時支援，三個月後轉正式回到國泰」（臨時支援，合理）
- 「職位設置有調整，最適合的位置仍在國泰」（組織調整，合理）

**紅旗** ❌：
- 「職位不太如預期」→ 暗示期望落差
- 「薪資沒談好」→ 薪資敏感，可能對職缺薪資也有期待
- 含糊其詞、閃躲 → 不信任信號

**記錄方式**：
```
台北富邦離職原因：___________________________
→ 合理性評估：🟢 合理 / 🟡 可接受 / 🔴 需觀察
```

### **Q2【薪資關鍵】根據妳的背景，薪資期待是多少？**

**背景**：
- 10 年金融系統開發
- 現職「經理」職稱
- 推估市場價 180-220 萬/年

**提問方式**（直接但友善）：
> 「根據妳 10 年的經驗和現在的職級，如果有合適的職缺，薪資的期待範圍大概是多少？給個範圍就可以，比如 X 到 Y 萬。」

**關鍵提示**：
- 不要問「最低願意接受多少」→ 容易被鎖定在低端
- 改問「期待範圍是多少」→ 了解真實市場認知

**好答案徵兆** ✅：
- 給出明確範圍：「180 到 220 萬」/ 「200 萬起」
- 有理由支撐：「根據現在的職級和經驗，我認為...」

**紅旗** ❌：
- 「還沒想好」→ 重視金錢的人會有明確想法
- 「看職缺」→ 不夠自信，容易被壓價
- 過高期待：> 250 萬 → CMoney 250 萬會不夠吸引

**記錄方式**：
```
薪資期待：_____________________________
→ CMoney 250萬 符合度：🟢 吻合 / 🟡 有談判空間 / 🔴 期待過高
→ Playsee 180萬 符合度：🟢 吻合 / 🟡 需談判 / 🔴 不符
```

### **Q3【到職時間】最快何時可以到職？**

**提問方式**：
> 「如果有合適的職缺，最快何時可以到職？目前公司是否需要交接期？」

**好答案徵兆** ✅：
- 「給現職 30 天 notice，預計 1.5 個月」（業界標準）

**紅旗** ❌：
- 「需要 3 個月以上交接」→ 職缺可能等不起
- 「有 non-compete」→ 法律風險

**記錄方式**：

### **Q4【技能驗證 - CQRS】玉山銀行核心系統轉型中，CQRS 設計的關鍵決策是什麼？**

**提問背景**：
- 履歷提到「導入 CQRS 架構模式，將原有多組分散 API，重構成以『設定檔』驅動的動態介接服務，最終將 API 收斂至單一查詢與單一異動入口」
- CMoney 強調「架構設計經驗，特別是大規模資料庫作業/高效能優化」

**詳細提問**：
> 「我看妳的履歷提到在玉山導入 CQRS 模式，將多組 API 收斂成單一查詢和異動入口。當時原有多少個 API？為什麼選擇 CQRS？在設計和實施過程中最大的困難是什麼？」

**希望聽到的內容** ✅：
- **API 複雜度**：原有 20-30+ 個 API，各自維護邏輯冗餘
- **為什麼選 CQRS**：讀寫分離、客製化需求零成本、長期維運成本降低
- **實施困難**：
  - 後端 team 習慣轉變（從單一服務 → 讀寫分離思維）
  - 設定檔驅動的動態邏輯如何測試
  - 性能監控如何調整
- **最終效果**：API 數量、響應時間、維運成本的改善數據

**紅旗** ❌：
- 只能說概念，沒有具體數字
- 「我們用了 CQRS 模式」但無法解釋為什麼
- 無法談論實施困難 → 可能沒真正主導

**記錄方式**：
```
CQRS 實務評估：
- 原有 API 複雜度：________________
- 設計決策邏輯：________________
- 實施困難與解決：________________
- 最終效果數據：________________

評估結論：🟢 深度可信 / 🟡 概念理解 / 🔴 只會說概念
```

### **Q6【高可用系統】Google Cloud Anthos 導入時最大的技術挑戰是什麼？**

**提問背景**：
- 履歷提到「主導網路銀行專案成功導入 Google Cloud Anthos 平台，負責改寫整合 CI/CD 流程並解決 JDK 版本相容性挑戰」
- CMoney 強調「分散式系統、高併發、架構設計」

**詳細提問**：
> 「Anthos 導入過程中最大的技術挑戰是什麼？當時妳是怎麼解決的？最後這個項目在系統穩定性或效能上有什麼改進？」

**希望聽到的內容** ✅：
- **具體技術挑戰**：
  - JDK 版本相容性（提到過）
  - 容器遷移的網路配置（pod-to-pod communication）
  - Kubernetes 資源分配與監控
  - 合規性要求（金融系統特殊要求）
- **解決過程**：不是「最後解決了」，而是「試過 A、失敗了、改用 B、成功」
- **效果驗證**：部署時間、宕機時間、擴展能力的改善

**紅旗** ❌：
- 「都是 Google 團隊幫我們做的」→ 自己的貢獻不清
- 只能說「成功導入」但無技術細節 → 可能只是參與者，非主導者
- 無法談及失敗或困難 → 可能只經歷過順利的部分

**記錄方式**：
```
Anthos 項目深度評估：
- 主要技術挑戰（3 個以上）：________________
- 解決方案（試錯過程）：________________
- 最終效果數據：________________
- 本人在其中的角色：________________

評估結論：🟢 深度主導 / 🟡 參與度中等 / 🔴 主要貢獻不清
```
