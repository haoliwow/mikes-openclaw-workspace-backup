# Java 工程師在做什麼 - 完整分解

---

## 一、通用 Java Backend 工程師（日常）

### 1. 系統設計 & 架構
- 根據業務需求設計 API 介面、數據模型、系統流程
- 決定技術棧選擇（框架、資料庫、緩存等）
- 做 code review、提升代碼質量

### 2. 功能開發
- 用 Spring Boot / Framework 寫業務邏輯
- 設計資料庫表結構、SQL 查詢優化
- 實現 REST API、WebSocket 等通信協議
- 寫單元測試、集成測試

### 3. 性能優化
- 識別系統瓶頸（CPU、記憶體、I/O）
- 優化 SQL 查詢、資料庫索引
- 實現緩存策略（Redis）
- 分析 JVM 調優（GC、heap 設置等）

### 4. 運維 & 監控
- 與 DevOps 配合進行 CI/CD 部署
- 讀日誌、調試線上問題
- 監控系統指標（吞吐量、響應時間、錯誤率）

### 5. 版本迭代
- 參與產品需求評估
- 規劃新功能、修復 bug
- 跟進客戶反饋

## 二、交易系統特定的 Java 工程師（一通場景）

### 核心差異：時間敏感 + 高併發 + 不能錯

### 1. Matching Engine / OrderBook 開發
**這是一通最核心的工作**

```
掛單流程：
Customer Order → Order Queue → Matching Engine → OrderBook → Trade Execution
  ↓
  Java Engineer 要搞定：
  • OrderBook 的數據結構設計（效率要高）
  • Matching 演算法（What if 這個訂單來晚了 10ms？）
  • Price-Time Priority 邏輯（公平性）
  • Partial Fill / Cancel 處理
```

- **必須理解**：
  - 訂單簿的 Best Bid/Ask 維護
  - 撮合規則（價格優先、時間優先、數量優先）
  - 分布式環境下的一致性問題

- **具體工作**：
  - 優化 OrderBook 查詢速度（毫秒級）
  - 設計高效的撮合演算法
  - 處理邊界情況（限價單、市價單、止損單）
  - 壓力測試（100K+ 訂單/秒時會發生什麼？）

### 2. 低延遲設計
**「從訂單到成交，要在 10ms 內完成」**

這不是做一般 CRUD 的延遲，而是交易的生死線。

- **要做的**：
  - 減少 GC pause（用適當的 JVM 設置）
  - 避免不必要的物件分配（對象池、無鎖設計）
  - 用 Lock-free 結構（ConcurrentHashMap, AtomicInteger）
  - 理解硬體級別（CPU cache line, 分支預測）
  
- **具體例子**：
  ```
  不行：每次訂單來都 new Order object
  要改：對象池 + 複用 order object
  
  不行：用 synchronized 鎖
  要改：用 CAS (Compare and Swap) 無鎖設計
  ```

### 3. 高併發 & 事件流處理
**「平時 1 萬單/秒，突然爆到 100 萬單/秒」**

- **用到的技術**：
  - Kafka：用來做 order flow 的事件隊列（持久化）
  - Thread pool / 非同步處理：避免阻塞
  - 背壓機制（backpressure）：當後端跟不上時怎麼辦
  
- **具體工作**：
  - 設計隊列策略（disruptor pattern 可能用到）
  - 監控隊列深度、延遲
  - 在高併發時保證消息不遺漏、不重複

### 4. 帳務 & 風控核心
**「每筆交易都要精確記錄，不能多不能少」**

- **要搞定的**：
  - 交易日誌（Audit Trail）：誰買了什麼、什麼時間、什麼價格
  - 資金帳務：買方扣錢、賣方加錢（兩邊必須一致）
  - 衝突解決：如果撮合引擎和帳務系統不同步怎麼辦
  
- **具體工作**：
  - 設計事務性（ACID）的帳務更新
  - 做數據對帳（Reconciliation）
  - 處理異常場景（partial trade、trade cancel）

### 5. 分散式系統設計
**「一個交易所有多個節點，怎麼確保一致性？」**

- **要考慮的**：
  - 副本（Replication）：交易數據要備份多份
  - 共識演算法：多個節點要同意一個交易結果
  - 故障轉移（Failover）：某個節點掛了，系統還能正常運行
  
- **具體工作**：
  - 用 ZooKeeper / Etcd 做協調
  - 設計 Leader-Follower 或 Raft 架構
  - 測試故障場景（一半節點掛掉時會怎樣）

## 三、具體日常場景（一通的 Java 工程師做什麼）

### 早上（09:00-11:00）
- 團隊 standup：誰在做什麼、有沒有 blocker
- Code review：檢查同事寫的 matching engine 優化
- 看 overnight 的系統日誌：有沒有異常

### 中午（11:00-13:00）
- **核心開發**：
  - 優化 OrderBook 的某個演算法
  - 或修復昨天發現的「高併發時偶現的 bug」
  - 或做壓力測試（1 百萬單/秒能不能扛）

### 下午（14:00-17:00）
- 性能分析：用 JProfiler 看哪裡 CPU 高
- 與 DevOps 溝通：新的監控指標怎麼埋點
- 與交易產品溝通：「我們能支持新的訂單類型嗎？」

### 晚上（17:00-19:00）
- 測試：寫自動化測試，確保新改的代碼不會破壞老功能
- 文檔：記錄 OrderBook 演算法的設計決策（為什麼選這個方案）
- 待命：如果是 on-call 的週，準備隨時處理線上問題

## 四、與其他角色的協作

### 與 DevOps / SRE 的協作
- DevOps：「系統現在 99.99% 可用性，夠不夠？」
- Java Engineer：「不夠，交易所需要 99.999%，我們要怎麼設計？」
- 結果：一起設計容錯機制、灾難恢復流程

### 與交易產品團隊的協作
- Product：「我們要支持新加坡交易所的訂單類型」
- Java Engineer：「好，但這會增加 Matching Engine 的複雜度，會影響延遲 +2ms」
- Product：「能接受，但其他優化要同時進行」

### 與 QA / 測試的協作
- QA：「我們怎麼測試交易系統在高併發時會不會丟單？」
- Java Engineer：「一起設計壓力測試框架，模擬 peak hour」

## 五、技能要求（分級）

### Level 1: 初階（2-3年）
- ✅ 能寫 Spring Boot 應用
- ✅ 懂 SQL 和基本的資料庫優化
- ✅ 能理解 REST API 設計
- ⚠️ 對 JVM 調優、分散式系統還不夠深

### Level 2: 中階（3-5年）
- ✅ 能獨立設計複雜系統的一部分
- ✅ 懂 Kafka、Redis、MongoDB 等技術棧
- ✅ 能分析性能問題，做優化
- ✅ 能參與架構決策
- ⚠️ 對超高併發、低延遲的細節優化還有成長空間

### Level 3: 資深（5年+，或交易系統背景）
- ✅ 能獨立擁有一個複雜系統的設計與優化
- ✅ 深入理解 JVM、硬體級別的性能特性
- ✅ 能做 Matching Engine、OrderBook 這類核心模組
- ✅ 能帶新人、做技術決策
- ✅ 對交易邏輯的理解能和交易員一樣深入

## 六、一通招一個資深 Java 工程師的期待

**現在缺什麼？**
- 能獨立 own Matching Engine 優化
- 理解低延遲系統的 trade-off（consistency vs performance）
- 有交易系統背景最好（不是絕對必要，但很加分）

**期待他/她的貢獻：**
1. **短期（3 個月）**：熟悉系統，提出 2-3 個性能優化方案
2. **中期（6-12 個月）**：lead 一個新功能的架構設計，例如支持 order types 擴展
3. **長期（1-2年）**：成為 matching engine 的 expert，教導團隊

**合作風格：**
- 不需要 micromanage，自主性強
- 可以和 product / trading team 直接對話
- 技術決策能參與

## 總結：一通的 Java 工程師 ≠ 一般的 Java 工程師

| 面向 | 一般企業 | 交易系統（一通） |
|------|---------|-----------------|
| **瓶頸** | 業務功能、可用性 | 延遲、吞吐量、一致性 |
| **難度** | 工程優化 | 工程 + 交易邏輯 + 數學 |
| **工具** | Spring Boot、MySQL | JVM 調優、Kafka、分散式算法 |
| **測試** | Unit Test、集成測試 | 壓力測試、混沌工程 |
| **成就感** | 「功能上線了」 | 「吞吐量從 50K 優化到 100K TPS」 |
