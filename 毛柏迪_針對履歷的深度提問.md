# 毛柏迪 針對履歷的深度提問清單

根據毛柏迪的具體履歷背景，以下是更深入、更針對性的提問內容。
這些問題不是在通用的 PM 篩選稿本中，而是專門針對他的經歷設計的。

## 📌 第一部分：現職深潛（戀結科技 PM）

### 戀結科技背景（STARTUP 背景核實）

**Q1：戀結科技的產品現狀**
```
「你說你主導產品從 0 到 1，到現在已經有 100+ 功能上線。
戀結科技現在的月活躍用戶（MAU）大概是多少？
轉換率或營收現狀怎麼樣？」

【目的】：
  - 驗證「100+ 功能」是否真的用戶需要，還是過度開發
  - 確認產品的市場驗證程度（是否達到 PMF - Product Market Fit）
  - 評估他的成功指標是否合理

【追問】：
「這 100+ 功能中，核心功能有多少個？用戶最常用的前 5 個功能是什麼？」
→ 了解他對產品優先級的理解
```

### 18 個月 Roadmap 的真實細節

**Q2：Roadmap 拆解**
```
「你說主導了 18 個月的 Roadmap，分成 9 個里程碑。
能描述一下這 9 個里程碑分別是什麼嗎？」

【目的】：
  - 驗證是否真的有系統的 Roadmap 規劃
  - 了解他的戰略思維（不只是時間拆解，而是價值拆解）
  - 看他是否能清晰表達

【評估重點】：
  ✅ 里程碑應該是「價值導向」而非「時間導向」
     例：「里程碑 1：驗證核心用戶痛點（3 個月）」
  ❌ 不應該是「月份 1-3、月份 4-6」這樣無意義的拆解

【追問】：
「9 個里程碑裡，有沒有因為市場反饋而調整過？」
→ 了解他的靈活性與應變能力
```

### 95% 準時交付率的背後

**Q3：交付率的計算方式**
```
「95% 準時交付率怎麼計算的？
這個比例是如何測量的？」

【目的】：
  - 驗證是否有明確的 deadline 制定機制
  - 了解他如何定義「準時」（延遲 1 天算不算失敗？）
  - 確認 5% 沒交付的里程碑發生了什麼

【評估重點】：
  ✅ 「我們定義了每個里程碑的硬截期，在 Jira 上追蹤。
      延遲超過 3 天算失敗。9 個里程碑中，有 1 個因為技術風險延遲，
      但我們快速調整計畫，最後在 deadline 前 2 周完成。」
  ❌ 「嗯...95% 就是大概準時啦」（無法量化）

【追問】：
「最大的一次 delay 是什麼原因？怎麼處理的？」
```

```
      估算了 TAM = X 億，SAM = Y 千萬，SOM = Z 百萬。

【追問】：
「根據這個 TAM/SAM 分析，你最後調整了什麼產品策略？」
```

### 商業模式設計的驗證

**Q6：6 種商業模式架構的決策過程**
```
「履歷提到『設計並驗證多元商業模式架構（含 B2B2C/B2B/B2C/C2C/C2B）』。
這 6 種模式最後選了哪一個？為什麼選那個？」

【目的】：
  - 驗證是否真的做過商業模式實驗
  - 了解他的決策邏輯（不只是想，而是真的驗證）
  - 看是否有量化的對比分析

【評估重點】：
  ✅ 「我測試了 3 種模式的 MVP：
      - B2C 訂閱制：用戶願意支付，但獲客成本高
      - B2B：企業客戶少，週期長
      - C2C 平台（最終選擇）：用戶既是供應方也是消費方，
        自動增長，我們只收取 20% 手續費。
      結果 C2C 的轉換率最高，所以選了這個。」
  ❌ 「我覺得 B2C 最簡單...」

【追問】：
「在你選擇 C2C 的過程中，有沒有遇過什麼難點？」
```

```
      我變成他的 mentor 而非 manager。現在他可以自己完成需求分析、
      優先級排序、跟工程協作的全流程。」
  ❌ 「就是他跟著我做了一年，然後會了」

【追問】：
「在培育的過程中，他最大的瓶頸是什麼？」
```

## 📌 第八部分：失敗與挑戰

## 📋 電話中的使用方式

1. **前 5 分鐘**：用通用的 Q1-6（邊界條件驗證）
2. **接下來 8 分鐘**：用 Q7-12（履歷深潛 - 現職、教育、獲獎）
3. **接下來 5 分鐘**：用 Q13-18（澳洲經驗、數據優化）
4. **最後 3 分鐘**：用 Q19-23（團隊、失敗、心理學應用）

**關鍵：** 這些問題不是每個都問，而是根據他的回答來決定要深潛哪些領域。
如果他在 Q7 的回答很棒，就跟進 Q8。如果回答很模糊，就換下一個領域。
